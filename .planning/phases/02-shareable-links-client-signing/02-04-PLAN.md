  <action>
    Create `src/components/ClientSignStep.tsx` — Step 5 of the wizard. This is the most complex client-facing component.

    **Props:** `onNext: () => void`, `onBack: () => void`, `agreementData: AgreementData`

    **Signature section (SIGN-02 + SIGN-03):**

    Two tabs: "Draw" and "Type" — equally presented, draw tab selected by default (per user decision).

    Tab bar: Two equal-width buttons at top. Active tab: forest-green background, white text. Inactive: white background, forest-green text, border.

    **Draw tab (SIGN-02):**
    - Reuse the canvas drawing pattern from the existing `SignaturePad.tsx` (NOT the component itself — build a new inline canvas here because the UX flow is different: no "Tap to Sign" placeholder, the canvas is always visible in the signing step)
    - Canvas: full width, 200px height, `touch-action: none` class, `passive: false` listeners for touchstart/touchmove/touchend
    - Helper baseline line near bottom of canvas
    - Clear button (eraser icon) in top-right corner
    - On save: `canvas.toDataURL('image/jpeg', 0.8)` (JPEG at 0.8 quality to reduce size — per research pitfall #4)
    - Store result in client sign store via `setDrawnSignature(dataUrl)`

    **Type tab (SIGN-03):**
    - Text input: "Type your full legal name"
    - Below the input: live preview of the typed name in Dancing Script font at 2rem, on a signature-line border
    - Store the typed name in the store via `setTypedSignatureName(name)` and `setSignatureType('type')`
    - When submitting, if signatureType is 'type', render the name to a hidden canvas using `ctx.fillText()` with Dancing Script font, then `toDataURL('image/jpeg', 0.8)` — this ensures all signatures are stored as image data consistently

    **Initials section (SIGN-04 per user decision: draw once, tap to apply):**

    Heading: "Your Initials"
    Subtext: "Draw your initials once below. Then tap the checkbox next to each section to confirm you've read and understood it."

    - Small canvas (full width, 100px height) for drawing initials — same touch handling pattern as signature canvas
    - Clear and save buttons for the initials canvas
    - Store result via `setDrawnInitials(dataUrl)`

    **Acknowledgment sections:**
    After initials are drawn, show the list of acknowledgment sections that need initialing:
    - Payment Terms acknowledgment
    - Insurance Disclosure acknowledgment
    - GPS Tracking acknowledgment
    - Geographic Restrictions acknowledgment
    - Unauthorized Recovery acknowledgment

    Each section: checkbox + acknowledgment text summary + small preview of the drawn initials (visible when checked). When checkbox is tapped, call `toggleAcknowledgment(sectionKey)` on the store. The initials image appears next to checked items as a small thumbnail.

    If initials haven't been drawn yet, disable the checkboxes and show "Please draw your initials above first" hint.

    **Validation before advancing:**
    - Signature must be provided (either drawn or typed — check drawnSignature or typedSignatureName is non-empty based on active tab)
    - Initials must be drawn
    - All 5 acknowledgment sections must be checked
    - If validation fails, show specific error: "Please provide your signature", "Please draw your initials", "Please acknowledge all sections"

    **Navigation:** "Back" text button and "Review & Submit" forest-green button. "Review & Submit" only enables when all validations pass.

    **Mobile UX (SIGN-05):**
    - Canvas containers have generous padding and clear visual boundaries
    - Touch targets are 44px minimum
    - Show a subtle hint: "Use your finger to sign" on mobile (detect via `window.matchMedia('(pointer: coarse)')`)
    - Canvas uses `touch-action: none` CSS AND `passive: false` event listeners to prevent scroll-while-drawing (critical for iOS — per research pitfall #1)
