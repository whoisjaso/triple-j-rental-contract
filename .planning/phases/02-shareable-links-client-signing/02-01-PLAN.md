    2. Create `supabase/migrations/002_client_sign_rls.sql` with TWO items:

       **a) Anon UPDATE policy** for the agreements table. The policy must:
       - Allow anon users to UPDATE agreements only when `token IS NOT NULL AND token_expires_at > now()`
       - Use both USING and WITH CHECK clauses with the same condition
       - This lets clients with a valid (non-expired) token update the `data` and `status` fields

       **b) BEFORE INSERT trigger on audit_log** that auto-populates `ip_address` and `user_agent` from PostgREST request headers when the inserted values are NULL. This is critical because browsers cannot self-report their real IP address, but PostgREST forwards the client IP via `x-forwarded-for` headers. The `get_client_ip()` and `get_user_agent()` functions already exist in 001_initial_schema.sql. The trigger ensures SIGN-06 and AUDIT-02 compliance (signature data stored with timestamp, IP, and user agent) without requiring any client-side code changes.

       ```sql
       -- Allow anonymous clients to update agreement data when they hold a valid, non-expired token
       CREATE POLICY "agreements_update_anon_by_token" ON agreements
         FOR UPDATE TO anon
         USING (token IS NOT NULL AND token_expires_at > now())
         WITH CHECK (token IS NOT NULL AND token_expires_at > now());

       -- Auto-populate ip_address and user_agent on audit_log inserts from PostgREST headers.
       -- Browsers cannot self-report IP, but PostgREST sets request.headers with x-forwarded-for.
       -- The get_client_ip() and get_user_agent() functions (from 001_initial_schema.sql) extract these.
       -- This trigger fires for ALL inserts (admin and client), filling NULLs transparently.
       CREATE OR REPLACE FUNCTION audit_log_auto_ip()
       RETURNS trigger AS $$
       BEGIN
         IF NEW.ip_address IS NULL THEN
           NEW.ip_address := get_client_ip();
         END IF;
         IF NEW.user_agent IS NULL THEN
           NEW.user_agent := get_user_agent();
         END IF;
         RETURN NEW;
       END;
       $$ LANGUAGE plpgsql;

       CREATE TRIGGER audit_log_auto_ip_trigger
         BEFORE INSERT ON audit_log
         FOR EACH ROW
         EXECUTE FUNCTION audit_log_auto_ip();
       ```

    1. **Add to `src/lib/agreements.ts`** -- two new exported functions:

       `generateShareLink(agreementId: string, expiryDays: number = 7)`:
       - Generate token via `crypto.randomUUID()`
       - Calculate `expiresAt` as current date + expiryDays
       - UPDATE the agreement: set `token`, `token_expires_at`, and `status: 'sent'`
       - INSERT an audit_log entry: action='sent', actor_type='admin', metadata with expires_at
       - Return `{ token, expiresAt: string }`

       `revokeLink(agreementId: string)`:
       - UPDATE the agreement: set `token: null`, `token_expires_at: null`, `status: 'draft'`
       - INSERT an audit_log entry: action='updated', actor_type='admin', metadata with `{ reason: 'link_revoked' }`
       - Setting token to NULL is critical -- the anon RLS policy `USING (token IS NOT NULL)` blocks all anon access to revoked agreements

    2. **Create `src/lib/clientSigning.ts`** -- new file with three exported functions using the anon Supabase client (import from `./supabase`):

       `fetchAgreementByToken(token: string)`:
       - SELECT id, agreement_number, status, data, token_expires_at from agreements WHERE token = token
       - Use `.single()` -- returns null on error (RLS blocks expired tokens automatically)
       - Return the row or null

       `recordClientView(agreementId: string)`:
       - INSERT into audit_log: agreement_id, action='viewed', actor_type='client', user_agent from `navigator.userAgent`, metadata with timestamp
       - Note: ip_address can be omitted from the INSERT -- the audit_log_auto_ip_trigger (from 002 migration) will auto-populate it from PostgREST request headers
       - Also UPDATE the agreement status to 'viewed' only if current status is 'sent' (don't overwrite 'signed'):
         ```typescript
         await supabase.from('agreements').update({ status: 'viewed' }).eq('id', agreementId).eq('status', 'sent')
         ```
       - Fire-and-forget (don't block on this)

       `submitClientSigning(token: string, clientData: Partial<AgreementData>, auditMeta: { userAgent: string })`:
       - Fetch current agreement by token to get existing data
       - Deep merge clientData into existing data (write a simple `deepMerge` helper or use spread for top-level sections -- the client updates renter, signatures, and acknowledgment sections only)
       - UPDATE agreement: set data to merged result, status to 'signed'
       - INSERT audit_log: action='signed', actor_type='client', user_agent, metadata with signed_at timestamp and fields_signed array
       - Note: ip_address is auto-populated by the database trigger -- do NOT attempt to capture or pass IP from client-side JavaScript
       - Throw on error

       Import `AgreementData` from `../types` for typing.
