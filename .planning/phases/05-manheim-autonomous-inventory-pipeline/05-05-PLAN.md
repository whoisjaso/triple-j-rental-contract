**Create `supabase/migrations/004_pipeline_cron.sql`:**

This migration sets up two things: (1) the pg_cron schedule and (2) enables the extensions needed.

```sql
-- Enable required extensions (pg_cron and pg_net are available on Supabase free tier)
CREATE EXTENSION IF NOT EXISTS pg_cron;
CREATE EXTENSION IF NOT EXISTS pg_net;

-- Store pipeline invocation credentials in Vault
-- NOTE: The actual secret values must be set BEFORE running this migration.
-- Run these manually in Supabase SQL Editor first:
--   select vault.create_secret('https://scgmpliwlfabnpygvbsy.supabase.co', 'project_url');
--   select vault.create_secret('YOUR_ANON_KEY', 'anon_key');
--   select vault.create_secret('YOUR_PIPELINE_SECRET', 'pipeline_secret');

-- Schedule the Manheim pipeline to run every 5 minutes
select cron.schedule(
  'manheim-pipeline-every-5min',
  '*/5 * * * *',
  $$
  select net.http_post(
    url := (select decrypted_secret from vault.decrypted_secrets where name = 'project_url')
           || '/functions/v1/manheim-pipeline',
    headers := jsonb_build_object(
      'Content-Type', 'application/json',
      'Authorization', 'Bearer ' || (select decrypted_secret from vault.decrypted_secrets where name = 'anon_key'),
      'x-pipeline-secret', (select decrypted_secret from vault.decrypted_secrets where name = 'pipeline_secret')
    ),
    body := '{"source":"pg_cron"}'::jsonb
  ) as request_id;
  $$
);
```

Also include SQL comments explaining:
- The Vault secrets must be set manually in Supabase SQL Editor before applying
- The pipeline_secret header prevents unauthorized external invocations
- pg_cron runs in the database context — it makes an HTTP call to the Edge Function
- The anon key in the Authorization header is the standard Supabase pattern for pg_cron -> Edge Function calls

Additionally, document the Supabase Storage bucket setup. Since bucket creation is typically done via the Supabase dashboard or CLI, add a SQL comment block:

```sql
-- MANUAL STEP: Create the 'vehicle-photos' storage bucket
-- Option A (Supabase Dashboard): Storage -> New Bucket -> Name: "vehicle-photos" -> Public: ON
-- Option B (Supabase CLI): Not supported for bucket creation
-- Option C (SQL):
INSERT INTO storage.buckets (id, name, public) VALUES ('vehicle-photos', 'vehicle-photos', true)
ON CONFLICT (id) DO NOTHING;

-- RLS policy for vehicle-photos bucket: allow anyone to read (public), allow authenticated to upload
CREATE POLICY "vehicle_photos_public_read" ON storage.objects FOR SELECT USING (bucket_id = 'vehicle-photos');
CREATE POLICY "vehicle_photos_anon_upload" ON storage.objects FOR INSERT WITH CHECK (bucket_id = 'vehicle-photos');
```

The bucket is public so photo URLs work without signed URLs (per RESEARCH.md anti-pattern: "do not store signed URLs — they expire").

Verify the complete Manheim autonomous inventory pipeline end-to-end. This task requires human verification because it involves external services (Gmail, Google Sheets, Twilio, Supabase deployment) that require credentials the human must configure.

**What was built across Plans 01-05:**

1. Edge Function (supabase/functions/manheim-pipeline/) processes three email types:
   - OVE Purchase Confirmation: parses HTML, decodes VIN via NHTSA, appends 26-column row to Google Sheet with "Incoming" status
   - Central Dispatch ACCEPTED: updates Towing Cost column for matching VINs
   - Manheim Sale Documents: extracts condition/title from PDF attachment, updates Sheet

2. SMS Notifications: admin receives text on new vehicle added or pipeline error (not for CD/Sale Docs updates)

3. Photo Upload Page (/photos): standalone guided photo capture with 7 shots, auto-publish on completion

4. Scheduling: pg_cron runs pipeline every 5 minutes with secret auth

5. Database: pipeline_log table for audit/dedup, vehicle-photos Storage bucket

**Pre-requisites (one-time setup):**
1. In Google Cloud Console: enable Gmail API + Sheets API, create OAuth2 credentials
2. In Google OAuth Playground: authorize with Gmail + Sheets scopes, get refresh token
3. Set Supabase Edge Function secrets:
   supabase secrets set GOOGLE_CLIENT_ID=xxx GOOGLE_CLIENT_SECRET=xxx GOOGLE_REFRESH_TOKEN=xxx PIPELINE_SECRET=xxx TWILIO_ACCOUNT_SID=xxx TWILIO_AUTH_TOKEN=xxx TWILIO_PHONE_NUMBER=xxx
4. Apply migrations: supabase db push (or run SQL in Supabase SQL Editor)
5. Set Vault secrets in Supabase SQL Editor (project_url, anon_key, pipeline_secret)
6. Deploy Edge Function: supabase functions deploy manheim-pipeline --no-verify-jwt

**Verification steps:**

A. Pipeline test (manual trigger):
   curl -X POST https://scgmpliwlfabnpygvbsy.supabase.co/functions/v1/manheim-pipeline -H "Authorization: Bearer YOUR_ANON_KEY" -H "x-pipeline-secret: YOUR_PIPELINE_SECRET" -H "Content-Type: application/json" -d '{"source":"manual-test"}'
